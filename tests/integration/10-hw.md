# Deferred physical confirmations — LED, button, mount

These require a human operator physically at the board (real K1 finger
press, real stick insertion, eyes on the real LED). They are **not**
blocking: `tests/unit/` (140/140, non-root) proves the parsing/
state-machine/argv-building logic; `tests/integration/hw_root.py` (6/6,
root on the box) proves the same production code drives the real Linux
mechanisms correctly — real loopback `mount(8)`/`umount(8)`, the real
`/sys/class/leds/user_led` sysfs attributes, and a real evdev fd via a
synthetic `uinput` device (including a genuine EBUSY umount retry against
a real held-open fd, and the *unfiltered* production
`button.resolve_k1_path()` against the box's real
`/proc/bus/input/devices`).
What remains below is specifically the part no loopback image, uinput
device, or fake sysfs directory can stand in for: a human, a real stick,
and a real finger.

Record date/operator/result inline as each item is run.

## LED — visual pattern confirmation

- [ ] `led.set('IDLE')` → LED visibly OFF.
- [ ] `led.set('VERIFYING')` → visibly slow (~0.5 Hz, "not yet") blink.
- [ ] `led.set('READY')` → visibly fast (~5 Hz) blink, and an operator
      unfamiliar with the system reliably identifies this as "act now"
      (the cold-human LED test).
- [ ] `led.set('RUNNING')` → LED visibly SOLID on (no blink).
- [ ] `led.set('ERROR')` → visibly rapid (~10 Hz) blink, clearly faster
      than READY.
- [ ] `led.set('SAFE_REMOVE')` → visibly slow, EVEN (~1 Hz) blink that
      *persists* (the existing tests only prove the sysfs attributes are set
      correctly and hold after the calling process exits — that the
      pattern is still correct-looking to a human, and that a cold
      operator reliably distinguishes it from READY/VERIFYING, is the
      part deferred here; the "persists across process death via
      RemainAfterExit + BindsTo" systemd-unit wiring itself is out of
      scope for this checklist).
- [ ] Ground-truth which physical LED is `user_led` vs `sys_led` on the
      actual board silkscreen/enclosure — needed for `README-OPERATOR.md`,
      not produced here.

## Button — real K1 press

- [ ] A real finger press-and-release of K1 within the window
      (`button.await_press(5)`) → returns `True`.
- [ ] Tape/hold K1 down BEFORE calling `await_press` → returns `False`
      immediately (anti-tamper). This task already proves the anti-tamper
      *mechanism* against a real evdev fd (a held synthetic uinput key,
      `hw_root.py` case (b)); what's deferred is the same result under an
      actual physical hold (finger or tape on the real switch).
- [ ] No press at all → `False` after the window elapses (real hardware,
      real timing — `hw_root.py` already proves this against a synthetic
      device; a physical run is still worth one confirmation pass).
- [ ] A press of the MASK/back button (adc-keys, code 158) DURING an
      `await_press` window → ignored; the window still times out `False`.
      Structurally guaranteed by evdev's per-device fd model (K1 and MASK
      are different `/dev/input/eventN` nodes; a MASK press cannot appear
      on the fd `await_press` is reading), not merely a hoped-for
      behavior — but worth one physical confirmation since a mislabeled
      button is exactly the failure mode worth guarding against.
- [ ] An operator who has never seen the board can correctly identify
      which physical button is K1 vs MASK (a photo/diagram identifying K1
      vs the do-not-press MASK button is `README-OPERATOR.md`'s job; this
      item is just the physical sanity check that the two buttons are in
      fact distinguishable in person before that doc is trusted to
      describe them).

## Mount — real USB stick

- [ ] Insert a real, `CA-XFER`-labeled FAT32 stick; `mount_ro(real_dev,
      mp)` (e.g. `/dev/sda1`, per the already-verified enumeration) succeeds;
      `mount | grep <mp>` shows `noexec,nosuid,nodev,ro`. (`hw_root.py`
      already proves this exact code path against a loopback vfat image;
      deferred here only because a *physical* stick is a different
      enumeration path — SuperSpeed USB-A → xHCI root hub → `usb-storage`
      → `sda1` — worth one confirmation with real hardware.)
- [ ] `umount` on the real stick returns 0; the OS reports it safe to
      physically remove only after that.
- [ ] Physically yank the stick mid-operation at least once (full lifecycle
      resilience is deferred until the harness/systemd unit exists —
      flagged here only as a reminder that the `umount` retry/backoff was
      validated against a *held-open-fd* EBUSY, not a
      *physically-removed-device* EBUSY/IO-error, which behaves
      differently at the block-device layer).

## Out of scope for this checklist

(USBGuard, udev, the systemd unit, dispatch), and `README-OPERATOR.md`
itself belong to later stages of the project — listed here only where
these modules are a direct dependency.
